# pairec4tigerllm 项目章程

## 1. 项目使命

**打通端到端的推荐流程，包含：召回 -> 精排 -> 重排。**

核心创新点：
- 将**生成式召回**引入到推荐系统中作为一路召回
- 使用 **Tenrec** 数据集的 `ctr_data_1M.csv` 进行训练和评估
- 基于 **pairec** 框架进行二次开发，最小化改动原则

## 2. 核心原则

### 2.1 打通流程优先

- 优先打通整个推荐流程，从用户发送请求到收到最终推荐产品
- 首要目标是**召回流程**的打通
- 其次是精排和重排流程的整合

### 2.2 简洁至上

- 所有功能都应保持**极简**
- 只保留核心功能，避免任何不必要的复杂性
- 代码清晰、易于理解和维护

### 2.3 方案开源

- 主体采用开源方案：
  - **pairec**：阿里巴巴开源的 Go 语言推荐框架
  - **RQ-VAE Recommender**：基于 PyTorch 的生成式推荐实现
- 避免使用闭源或商业组件

### 2.4 代码规范

- **Python**：严格遵循 PEP 8 规范
- **Go**：使用标准格式化工具 gofmt
- 所有代码必须包含清晰的注释和文档字符串
- 公共 API 必须有完整的文档说明

## 3. 技术约束

### 3.1 后端语言

- **主要语言**：Python（模型训练）+ Go（服务部署）
- **数据层**：Python 负责数据处理和模型训练
- **服务层**：Go 负责在线推理服务

### 3.2 模型训练

- **硬件要求**：使用 GPU 进行训练
- **训练框架**：PyTorch
- **数据**：仅使用 Tenrec 的 `ctr_data_1M.csv`，禁止私自构造数据

### 3.3 生成式模型部署

- **优先方案**：使用 **TensorRT-LLM** 进行模型部署
- **推理优化**：支持 FP16/INT8 量化
- **服务接口**：支持 HTTP/gRPC

### 3.4 模型架构

- **RQ-VAE**：将物品编码为语义 ID
  - 4 层量化器
  - 每层 256 个码本向量
  - 输出 4 维语义 ID

- **GPT2 Decoder**：生成式推荐模型
  - Decoder-only 架构
  - 6 层 Transformer
  - 8 头注意力
  - 自回归生成

## 4. 禁止事项

### 4.1 虚假实现

**严格禁止**引入任何形式的虚假服务或 Mock 数据。

- 所有数据必须来自 **Tenrec 数据集**
- 不允许使用随机生成或硬编码的推荐结果
- 不允许使用占位符服务替代真实模型推理

### 4.2 数据构造

- **禁止**私自构造训练数据
- **禁止**修改原始数据集的内容或分布
- 只允许进行必要的预处理（ID 映射、序列化等）

### 4.3 过度工程

- **禁止**引入不必要的抽象层
- **禁止**过度设计架构
- 保持代码简单直接

## 5. 项目结构约定

### 5.1 目录结构

```
pairec4tigerllm/
├── data/           # 数据处理（Python）
├── training/       # 模型训练（Python）
├── inference/      # 推理服务（Python）
├── services/       # pairec 服务（Go）
├── configs/        # 配置文件
├── scripts/        # 工具脚本
└── tests/          # 测试用例
```

### 5.2 文件命名

- Python 文件：蛇形命名法（`data_loader.py`）
- Go 文件：蛇形命名法（`generative_recall.go`）
- 配置文件：使用描述性名称（`pairec_config.json`）

### 5.3 代码组织

- 每个模块必须有 `__init__.py`（Python）
- 公共 API 必须在 `__init__.py` 中导出
- 配置和实现分离

## 6. API 规范

### 6.1 请求格式

```bash
POST /api/rec/feed
Content-Type: application/json

{
    "uid": "76295990",
    "size": 10,
    "scene_id": "home_feed"
}
```

### 6.2 响应格式

```json
{
    "code": 200,
    "msg": "success",
    "request_id": "e332fe9c-7d99-45a8-a047-bc7ec33d07f6",
    "size": 10,
    "experiment_id": "",
    "items": [
        {
            "item_id": "248791390",
            "score": 0.9991594902203332,
            "retrieve_id": "generative_recall"
        }
    ]
}
```

### 6.3 错误处理

- HTTP 状态码：200（成功）或 500（错误）
- 错误信息在 `msg` 字段中
- 必须包含 `request_id` 用于追踪

## 7. 开发流程

### 7.1 里程碑

1. **M1**：RQ-VAE 和 Decoder 模型训练完成
2. **M2**：TensorRT-LLM 推理服务部署完成
3. **M3**：pairec 集成完成，服务可运行
4. **M4**：测试通过，文档完善

### 7.2 提交流程

- 每个功能模块完成后进行代码审查
- 关键路径上的代码需要单元测试覆盖
- 集成测试通过后才能合并

## 8. 性能要求

### 8.1 延迟要求

- P99 延迟 < 100ms（召回阶段）
- TensorRT-LLM 推理 < 50ms

### 8.2 吞吐要求

- 单节点 QPS > 1000
- 支持批量推理优化
